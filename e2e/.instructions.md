# End-to-End Testing Guidelines

This applies to all E2E test code in the `e2e/` directory.

## Testing Framework

- **Framework**: Playwright
- **Language**: TypeScript
- **Page Objects**: Pattern-based organization
- **See**: `docs/UI-tests.md` for comprehensive testing guide

## File Organization

### `/e2e` Structure

```
e2e/
├── page-objects/          # Page object classes
├── src/                   # Test utilities and setup
├── tests/                 # Actual test files
├── infrastructure/        # Environment and database setup
├── resources/            # Test data and fixtures
├── playwright.config.ts  # Playwright configuration
└── tsconfig.json        # TypeScript config
```

## Page Object Pattern

### Purpose

- Encapsulate selectors and interactions for UI elements
- Make tests maintainable when UI changes
- Promote code reuse across tests

### Example Page Object

```typescript
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.locator('input[type="email"]');
    this.passwordInput = page.locator('input[type="password"]');
    this.loginButton = page.locator('button:has-text("Login")');
    this.errorMessage = page.locator('[role="alert"]');
  }

  async goto() {
    await this.page.goto('/login');
    await this.page.waitForLoadState('networkidle');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
    await this.page.waitForLoadState('networkidle');
  }

  async getErrorMessage(): Promise<string> {
    return await this.errorMessage.textContent();
  }
}
```

## Writing Tests

### Test Structure

```typescript
import { test, expect } from '@playwright/test';
import { LoginPage } from '../page-objects/LoginPage';

test.describe('User Authentication', () => {
  test.beforeEach(async ({ page }) => {
    // Setup before each test
  });

  test('should login with valid credentials', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login('user@example.com', 'password');

    await expect(page).toHaveURL('/dashboard');
  });

  test('should show error with invalid credentials', async ({ page }) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login('user@example.com', 'wrongpassword');

    const error = await loginPage.getErrorMessage();
    expect(error).toContain('Invalid credentials');
  });
});
```

### Best Practices

- **One action per assertion** when possible
- **Descriptive test names** clearly stating what is being tested
- **Use fixtures** for common setup (authentication, data)
- **Wait for conditions** instead of fixed delays
- **Use meaningful selectors** (role, text, testid)

## Selectors Strategy

### Preferred Selectors (in order)

1. **Role-based**: `[role="button"]`, `[role="heading"]`
2. **Text**: `button:has-text("Save")`
3. **Test ID**: `[data-testid="submit-button"]`
4. **CSS/XPath**: Last resort, more brittle

### Example

```typescript
// Good
await page.locator('[role="button"]:has-text("Submit")').click();

// Acceptable
await page.locator('[data-testid="submit-btn"]').click();

// Avoid
await page.locator('div > form > button:nth-child(3)').click();
```

## Test Data Management

### Using Fixtures

```typescript
test.beforeEach(async ({ page }) => {
  // Create test data
  const testUser = { email: 'test@example.com', password: 'Test123!' };
  // Store in context for test
  test.info().annotations.push({ type: 'user', description: testUser.email });
});
```

### Resource Files

- Store test data in `resources/` directory
- Use JSON or TypeScript files for structured data
- Import into tests as needed

```typescript
import { testUsers } from '../resources/testUsers';

const user = testUsers.adminUser;
```

## Handling Dynamic Content

### Wait Strategies

```typescript
// Wait for network to settle
await page.waitForLoadState('networkidle');

// Wait for specific element
await page.locator('.data-table').waitFor();

// Wait for navigation
await page.waitForNavigation();

// Wait with timeout
await page.locator('.dialog').waitFor({ timeout: 5000 });
```

### Handling Flakiness

- Use explicit waits instead of `page.waitForTimeout()`
- Ensure elements are visible/enabled before interaction
- Use `waitForLoadState('networkidle')` for AJAX requests

## Performance Considerations

- Parallel test execution by default
- Tests should be independent
- Use `test.skip()` or `test.only()` for filtering
- Monitor test execution time

## Common Patterns to Avoid

- ❌ Hard-coded delays (`await page.waitForTimeout(1000)`)
- ❌ Tests that depend on execution order
- ❌ Using `page.evaluate()` for interactions (use Playwright APIs)
- ❌ Brittle XPath/CSS selectors tied to HTML structure
- ❌ Tests without proper error messages
- ❌ Mixing concerns (UI + API testing in same test)
