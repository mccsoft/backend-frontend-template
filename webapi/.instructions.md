# Backend Development Guidelines

This applies to all backend code in the `webapi/` directory.

## Technology Stack
- **Framework**: ASP.NET Core
- **Language**: C# (.NET 10)
- **ORM**: Entity Framework Core
- **Authentication**: OpenIddict (OpenID Connect)
- **Background Jobs**: Hangfire
- **Logging**: Serilog
- **API Documentation**: Swagger/OpenAPI

## Key Patterns

### Architecture Layers
1. **Domain** - Domain entities, value objects, interfaces (no EF dependencies)
2. **Application** - Controllers, DTOs, services orchestration
3. **Infrastructure** - Database context (DbContext), repositories
4. **API** - REST endpoints, middleware, configuration

### Nullable Reference Types
- **Enabled by default** (all projects have `<Nullable>annotations</Nullable>`)
- All reference types must be explicitly marked as nullable (`?`)
- Use `#nullable enable/#nullable disable` per-file if needed

### Naming Conventions

- **Namespaces**: `MccSoft.TemplateApp.{Layer}.{Feature}`
  - Example: `MccSoft.TemplateApp.App.Features.Users`

- **Classes**:
  - Entities: Singular noun (`User`, `Order`)
  - Services: Verb suffix (`UserService`, `OrderProcessor`)
  - Controllers: `{Feature}Controller` (`UsersController`)
  - DTOs: `{Entity}Dto` (`UserDto`)
  - Requests/Responses: `{Action}{Entity}{Request|Response}`

- **Methods**: PascalCase, descriptive (`GetUserById`, `CreateUserAsync`)

- **Private Fields**: camelCase with underscore (`_userRepository`)

- **Constants**: UPPER_SNAKE_CASE

## Entity Framework Core

### DbContext Usage
- Single DbContext per application
- Located in `Database` project
- Use `DbSet<T>` for entity collections
- Enable query logging for debugging

### Migrations
```bash
# In webapi directory
dotnet ef migrations add MigrationName --project src/MccSoft.TemplateApp.Database
dotnet ef database update --project src/MccSoft.TemplateApp.Database
```

### Entity Definition Best Practices
- Use value objects for complex types
- Implement domain events in entities
- Keep entities focused on domain logic, not persistence concerns
- Use data annotations minimally; prefer Fluent API in ModelBuilder

### Example Entity
```csharp
#nullable enable
namespace MccSoft.TemplateApp.Domain.Features.Users;

public class User
{
    public long Id { get; set; }
    public required string Email { get; set; }
    public string? FullName { get; set; }
    public bool IsActive { get; set; } = true;
    
    public void Deactivate()
    {
        IsActive = false;
        AddDomainEvent(new UserDeactivatedEvent(Id, Email));
    }
    
    private List<IDomainEvent> _domainEvents = [];
    public IReadOnlyList<IDomainEvent> DomainEvents => _domainEvents.AsReadOnly();
    
    public void AddDomainEvent(IDomainEvent @event) => _domainEvents.Add(@event);
}
#nullable restore
```

## REST API Guidelines

### HTTP Methods
- `POST` - Create new resources
- `GET` - Retrieve resources
- `PATCH` - Partial updates (preferred over PUT)
- `DELETE` - Remove resources

### Status Codes
- `200 OK` - Successful GET, PATCH, DELETE
- `201 Created` - Successful POST (include Location header)
- `204 No Content` - Successful update with no response body
- `400 Bad Request` - Invalid input
- `401 Unauthorized` - Authentication required
- `403 Forbidden` - Authenticated but not authorized
- `404 Not Found` - Resource not found
- `409 Conflict` - Resource conflict (e.g., duplicate)
- `500 Internal Server Error` - Server error

### Endpoint Examples
```csharp
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<ActionResult<UserDto>> GetById(long id) { }
    
    [HttpPost]
    public async Task<ActionResult<UserDto>> Create(CreateUserRequest request) { }
    
    [HttpPatch("{id}")]
    public async Task<IActionResult> Update(long id, UpdateUserRequest request) { }
    
    [HttpDelete("{id}")]
    public async Task<IActionResult> Delete(long id) { }
}
```

## Async/Await

- **Always use `async/await`** for I/O operations
- Return `Task` or `Task<T>` from async methods
- Suffix async methods with `Async` (optional but recommended)
- Use `ConfigureAwait(false)` in library code

```csharp
public async Task<User?> GetUserByIdAsync(long userId)
{
    return await _context.Users.FirstOrDefaultAsync(u => u.Id == userId);
}
```

## Dependency Injection

- Configure in `Program.cs` or Startup configuration
- Use constructor injection
- Scope: Scoped for DbContext, Transient for stateless services, Singleton for heavy instances

```csharp
public class UserService
{
    private readonly IRepository<User> _userRepository;
    private readonly ILogger<UserService> _logger;
    
    public UserService(IRepository<User> userRepository, ILogger<UserService> logger)
    {
        _userRepository = userRepository;
        _logger = logger;
    }
}
```

## Logging with Serilog

- Injected as `ILogger<T>` or `ILogger`
- Use log levels appropriately: Debug, Information, Warning, Error, Fatal
- Include contextual information

```csharp
_logger.LogInformation("User {UserId} created successfully", userId);
_logger.LogError(ex, "Failed to process user {UserId}", userId);
```

## Background Jobs with Hangfire

- Configure job storage (SQL Server or other)
- Use `IBackgroundJobClient` for job scheduling
- Implement recurring jobs with `RecurringJob`

```csharp
public class UserCleanupService
{
    public void ScheduleCleanup()
    {
        RecurringJob.AddOrUpdate<UserCleanupJob>(
            "cleanup-inactive-users",
            x => x.Execute(),
            Cron.Daily);
    }
}
```

## Error Handling

- Use specific exception types
- Catch exceptions at appropriate boundary (controller level)
- Return proper HTTP status codes
- Log exceptions with context

```csharp
try
{
    return await _userService.CreateUserAsync(request);
}
catch (InvalidOperationException ex)
{
    _logger.LogWarning(ex, "Invalid user creation request");
    return BadRequest(new { error = ex.Message });
}
catch (Exception ex)
{
    _logger.LogError(ex, "Unexpected error creating user");
    return StatusCode(500, new { error = "An unexpected error occurred" });
}
```

## Data Transfer Objects (DTOs)

- Separate domain entities from API contracts
- Use AutoMapper or manual mapping
- Request: `{Entity}Request` or `Create{Entity}Request`
- Response: `{Entity}Dto`

```csharp
public class CreateUserRequest
{
    public required string Email { get; set; }
    public string? FullName { get; set; }
}

public class UserDto
{
    public long Id { get; set; }
    public string Email { get; set; }
    public string? FullName { get; set; }
    public bool IsActive { get; set; }
}
```

## Testing

## Performance Considerations

- Use `AsNoTracking()` for read-only queries
- Select only needed columns with `Select()`
- Use pagination for large result sets
- Implement caching where appropriate
- Monitor slow queries in logs

## Package Management

- Centralized versions in `Directory.Packages.props`
- Add packages there, then reference in `.csproj` without version
- Keep dependencies minimal and up-to-date

## Common Patterns to Avoid

- ❌ Exposing DbContext outside data layer
- ❌ Blocking calls (`.Result`, `.Wait()`)
- ❌ Catching generic `Exception` unnecessarily
- ❌ Mixing concerns (domain logic in controllers)
- ❌ N+1 query problems (use `Include()` for related entities)
- ❌ Storing secrets in code
